# Vispeak — План разработки

> Обновляется каждым этапом. КОММИТИТСЯ в репозиторий (НЕ в .gitignore) —
> синхронизация между машинами. Прошлый рассинхрон случился из-за игнора.
> Язык: русский (проза и решения), технические идентификаторы/код/пути/лог-метки —
> в оригинале (англ.).

Vispeak — desktop-приложение для локальной офлайн-транскрибации
голоса по глобальной горячей клавише (аналог SuperWhisper). Windows-only.
Разработка ведётся промтами через агента (Antigravity IDE). Проект начат с нуля.

---

## Стек технологий
- **Backend**: Tauri 2 + Rust
- **Frontend**: React + TypeScript + Tailwind CSS (v4), npm
- **Аудио**: `cpal` (запись), `rubato` -> кастомный ресемплер для 16 kHz
- **VAD**: Silero VAD (ONNX, 2.2 МБ, встроена в бинарник)
- **Движки транскрибации** (trait TranscriptionEngine): `whisper-rs` (Whisper cpp,
  GGML) + `transcribe-rs` (ONNX: Parakeet, GigaAM, Canary) + `transcribe.cpp`
  (ggml: Nemotron, Qwen — статус имплементации уточняется)
- **Симуляция ввода**: `enigo` (вставка через буфер + Ctrl+V)
- **Хоткеи**: `rdev` (низкоуровневый системный хук WH_KEYBOARD_LL, левые/правые
  модификаторы, одиночные модификаторы)
- **Прочее**: rusqlite (история), rodio (звуковые сигналы), Windows CoreAudio /
  WASAPI (ducking)

## Инварианты (нарушать нельзя)
- Оверлей НИКОГДА не берёт фокус ввода (`focus: false`, проверка Блокнотом).
- Оверлей всегда тёмный (тема приложения на него не влияет).
- Аудио строго mono 16кГц; транскрибация в отдельном потоке (не блокирует UI).
- Хоткей работает при свёрнутом окне.
- duck/restore строго парные (guard / Drop-семантика).
- Дизайн только через семантические токены (никаких хардкод-цветов).
- Все строки UI через i18n (ru/en); из Rust — коды ошибок, не тексты.
- Новые поля настроек — только с `#[serde(default)]` (обратная совместимость).
- Позиционирование окон — в физических пикселях с учётом scale_factor монитора.

## Архитектура (цикл диктовки)
1. **Глобальный хоткей**: слушаем клавиатуру через `rdev` в отдельном потоке.
   Режимы Push-to-Talk и Toggle, различение левых/правых модификаторов. Дефолт —
   Ctrl+Space. PTT <300мс удержания — тихая отмена; toggle-запись <0.5с ->
   «речь не распознана». Esc — глобальная отмена (регистрируется на время цикла).
2. **Запись**: через `cpal` старт записи с микрофона. Программное усиление (Gain)
   с клиппингом (защита от перегруза). Показ оверлея («Запись...»).
3. **Обработка**: по отпусканию хоткея аудио -> Silero VAD (обрезка тишины по
   краям) -> whisper-rs / transcribe-rs. Модель загружена в память заранее,
   транскрибация в отдельном потоке.
4. **Вставка** (порядок СТРОГО зафиксирован):
   `запись текста в буфер` (для копирования/вставки или при policy=keep)
   -> симуляция `Ctrl+V` (enigo)
   -> пауза на обработку вставки (50 мс)
   -> `finalize_clipboard(policy)` (restore: восстановление старого буфера ~300мс;
      keep: верификация с повторными попытками)
   -> автоотправка Enter (`Key::Return` / `Ctrl+Return`) строго ПОСЛЕДНЕЙ.
   Оверлей скрывается.

## Оверлей (3 скина)
Переключается настройкой `overlay_skin` ("full" | "compact" | "mini"). Все скины
используют общий хук `useOverlayState`. При смене настройки размер и положение окна
пересчитываются в `src-tauri/src/lib.rs` перед показом.

- **Полный (`OverlayFull`)**: ~400x160 px. Шапка (целевое приложение), текстовые
  подсказки, иконка приложения, эквалайзер, «Esc — отмена». Свечение состояний:
  запись #ff5533, обработка cyan, успех зелёный (+превью), ошибка красный.
- **Компактный (`OverlayCompact`)**: ~340x90 px. Капсула в одну строку с
  укороченной волной, без шапки. Обработка — три пульсирующие точки + «Обработка...»;
  успех — галочка (~700мс); ошибка — крестик+текст; «Скопировано» в режиме «только
  копировать».
- **Мини (`OverlayMini`)**: ~96x34 px (~144x82 логическое окно с учётом glow).
  Капсула-пилюля «У курсора» строчной высоты: запись — эквалайзер на всю ширину
  (13 точек высотой до 22 px); обработка — три пульсирующие точки; успех — галочка
  (~16 px, ~600 мс); ошибка — крестик (~16 px).

**Осознанное решение (тёмная тема оверлея)**: все скины индикатора ОСТАЮТСЯ
ТЁМНЫМИ независимо от темы приложения (Системная/Тёмная/Светлая). Капсула поверх
произвольных окон — тёмная с цветным свечением читается универсально, цвета
состояний (красный/cyan/зелёный) — часть языка продукта.

### Положение оверлея
Зависит от скина и настройки `overlay_position`.
1. **"full" и "compact" (или при сбое определения курсора)**: универсальная
   якорная модель относительно `rcWork` целевого монитора (Низ по центру / Правый
   нижний угол / Левый нижний угол / Верх по центру).
2. **"mini" ("У курсора")**: в `start_recording` вызывается
   `caret_position::get_caret_position` с таймаутом 100 мс: координаты каретки через
   `IUIAutomation` (сначала `TextPattern2.GetCaretRange`, затем
   `TextPattern.GetSelection`), при неудаче фоллбек на `GetGUIThreadInfo`
   (`rcCaret`).
3. **Валидация и физические координаты**: любые координаты проходят
   `validate_caret_rect`: разумные размеры (`0 < height <= 3000 px`,
   `0 <= width <= 5000`), внутри экрана монитора (`MonitorFromRect`), внутри
   видимого окна приложения `GetWindowRect(target_hwnd)` с допуском `±64 px` на
   тени/рамки. Позиционирование — `tauri::Position::Physical(PhysicalPosition)` с
   точным учётом `scale_factor` каждого монитора.
4. **Классификация `CaretKind` и единое правило**:
   - **`Caret`**: узкий прямоугольник (`width <= 4` ИЛИ `height <= 40 && width <=
     height * 2`).
   - **`Field`**: небольшой ввод (`width <= 600 px` И `height <= 60 px`) — поиск,
     адресная строка, чат.
   - **`Area`**: больше (`width > 600 px` ИЛИ `height > 60 px`) — многострочные
     редакторы (VS Code, Monaco, большие `textarea`).
   - **Горизонталь (X)**: точка привязки — левый край прямоугольника (`rect.left`)
     для всех типов; капсула ЦЕНТРИРУЕТСЯ на этой точке; затем горизонтальный кламп
     к `rcWork` с отступом 8 px.
   - **Вертикаль (Y) — ВСЕГДА НАД**: низ видимого края капсулы на 8 px выше верха
     прямоугольника. Исключение: если верх капсулы выходит за `rcWork.top` — капсула
     ПОД (верх капсулы на 8 px ниже низа прямоугольника).
   - **`Area` + мышь**: если курсор мыши ВНУТРИ прямоугольника Area, `pt.y` мыши
     используется только для ВЕРТИКАЛИ (капсула над нужной строкой); по горизонтали
     — общее правило (левый край). Мышь вне Area — чистый фоллбек на якорь
     (`anchor_fallback`).
   - **`GLOW_PADDING`**: все отступы (8 px) от ВИДИМОГО края капсулы с компенсацией
     прозрачного поля свечения (`phys_glow`).

### Тестовая матрица позиционирования "У курсора" (~96x34 px)
| Контекст | Метод | CaretKind | above/below | Поведение |
|---|---|---|---|---|

## Design decisions & history (почему так, что отвергнуто)
- Оверлей: glassmorphism -> near-black минимализм (сознательный разворот).
- Тема оверлея отделена от темы приложения; оверлей всегда тёмный (инвариант).
- Мини-скин: горизонталь центрируется на левом крае строки, вертикаль всегда над —
  выбрано после ~6 итераций других вариантов.
- История: остановились на off/5/10 вместо раздельных политик аудио/текст.
- Буфер: «Убирать после вставки» невозможно сделать «вообще не писать в буфер» для
  clipboard-способов (Ctrl+V читает из буфера); след в истории Windows (Win+V) —
  принятое ограничение. Не пишущий в буфер способ — только посимвольный enigo.
- Референсы дизайна выбирались по сгенерированным изображениям и скринам реальных
  приложений (Spokenly, Raycast).
- Иконка: убрана квадратная подложка и внешнее свечение, осталось только облако.
- Эквалайзер оверлея: убрана искусственная зеркальная купольная огибающая ("горка"); переведено на 60 FPS rAF с асимметричным сглаживанием (fast attack / slow decay) и фазовыми сдвигами полос.

## Bug Fixes & Stability (история решённых багов)

### Caret-позиционирование в Electron (VS Code, Slack, Antigravity)
Обход UIA-бага с огромными зонами (degenerate text selection): отказ от
`ExpandToEnclosingUnit(TextUnit_Character)`; валидация габаритов
(`FIELD_MAX_WIDTH = 4000`, классификация Field); приоритет точного bounding rect;
фикс anomaly detection в lib.rs (кламп X без ложных аномалий). Улучшено логирование
`caret_trace`, чистка неиспользуемых переменных/импортов в caret_position.rs и lib.rs.

### Сбой позиционирования "У курсора" (отрицательный top / дрифт при DPI > 100%)
Причина: некорректный `hwndFocus` при нулевом `hwndCaret` в `GetGUIThreadInfo` +
передача физических координат в логический `set_position(Position::Logical)`, где
Tauri повторно умножал на scale_factor. Решено: строгая валидация
`validate_caret_rect` (высота, `rcMonitor`, `GetWindowRect`) + переход на
`Position::Physical(PhysicalPosition)` с учётом scale_factor монитора.

### Фризы ввода ("slideshow mouse")
Устранена утечка зомби-rdev-потоков; heartbeat обновляется на ВСЕ события (включая
мышь); callback хука переведён в lock-free минимум с делегированием обработки
(`prune_stuck_keys`, валидация хоткея) в фоновый воркер.

### Защита хоткея от скриншотеров/перехватчиков (`hotkeys::setup_hotkeys`)
Сторонние скриншотеры (Snipping Tool `Win+Shift+S`, ShareX, Lightshot `Alt+S`)
ломают хук `WH_KEYBOARD_LL` (`rdev::listen`) двумя способами:
- **Case B — залипание клавиш**: скриншотер перехватывает KeyRelease обычной
  клавиши/модификатора. Решено: `is_key_physically_pressed` расширена на ВСЕ клавиши
  (`VK_A..VK_Z`, `0..9`, `F1..F12`, `VK_BACK/RETURN/TAB`, модификаторы) через
  `GetAsyncKeyState` в `prune_stuck_keys()` (перед каждым событием и по таймеру раз
  в 2.5 с, лог `[debug][hotkeys] Self-correcting stuck key: ...`). Защита в глубину:
  `is_hotkey_pressed` требует все клавиши хоткея и отсутствие посторонних
  модификаторов, но игнорирует посторонние обычные клавиши (залипшая S не блокирует
  Right Ctrl; Ctrl+Shift+X не срабатывает от хоткея Ctrl+X).
- **Case A — снятие хука системой**: при блокировке очереди сообщений Windows молча
  отключает хук. Решено Watchdog-таймером: при тишине хука (`heartbeat > 5s`) и
  активном вводе (`GetLastInputInfo`) выдерживается пауза 600 мс (защита от ложных
  перезапусков при выходе из простоя); если событий от rdev так и нет — инкремент
  `LISTENER_GEN` и перезапуск `rdev::listen` в новом чистом потоке (generation guard:
  устаревший listener становится пустышкой). Debug-лог зажат (обычные клавиши не
  логируются — приватность).

### Буфер обмена — самотесты и инварианты (`finalize_clipboard`, `paste.rs`)
Повторная запись распознанного текста (re-check) разрешена СТРОГО внутри ветки
`policy == "keep"`. Ветка `restore` выполняет только восстановление `previous_text`
(до 15 попыток за 450 мс, защита от блокировки буфера целевым окном после Ctrl+V).
Был маятниковый баг keep<->restore, лечили дважды. Любые правки в paste.rs
верифицируются интеграционными самотестами `cargo test --lib paste::tests`
(проверка через 1.5 с на реальном системном буфере, синхронизация через
`CLIPBOARD_MUTEX`).

### Аудио: зомби-стримы, ducking, watchdog (`audio::run_audio_session`, `ducking`)
- **Зомби-стрим и мёртвый ресемплер**: счётчик сырых сэмплов рос в фоне вне записи,
  ресемплер выдавал 0 накопленных. Причина: гонка потоков в
  `start_recording`/`start_preview`. Решено: атомарная остановка предыдущего канала
  (`stop_tx.take().send(())`) и установка флагов (`is_recording`, `stop_tx`) строго
  внутри единого Mutex-лока до запуска нового потока; 100% изоляция состояния
  (`NearestResampler` и буфер сбора создаются локально в каждом короткоживущем
  `worker_process`). Если сэмплов 0 или разрыв >500 мс — Watchdog.
- **Ducking: двойное снижение и восстановление после сбоя** (`DuckingGuard`,
  `ducked_volumes.json`): при перекрывающихся циклах громкость сторонних приложений
  (VLC) снижалась повторно и оставалась приглушённой. Решено: флаг-гард `IS_DUCKED`
  против повторного приглушения; снимок исходных уровней `ORIGINAL_VOLUMES` держится
  в памяти И синхронно сохраняется в `%APPDATA%/Vispeak/ducked_volumes.json`. При
  старте (`restore_all_on_startup` в lib.rs) и при завершении/сбое
  (`DuckingGuard::drop`) все сессии возвращаются к исходному уровню, файл снимка
  удаляется.
